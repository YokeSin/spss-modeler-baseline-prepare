# セクション別 Markdown スニペット例

[phase2-summary-report.md](../phase2-summary-report.md) で生成する `<RUN_DIR>/reports/summary_report.md` の各章に貼る **構造の見本**。

**このファイルの中身を summary_report.md に丸ごとコピーしないこと。** 必要なスニペットだけ抜き出して、`<...>` のプレースホルダーを `outputs/` 配下の HTML から読み取った実値で置き換えてから貼る。

数値・文字列はすべて出力 HTML から読み取った実値で書く。読めなかった値は `-` または `(取得失敗)` と明示する。

## プレースホルダー凡例

スキル内で使う `<...>` 記法は「実値に必ず置換するマーカー」。phase2 が禁止しているのは `{{...}}` 形式のみで、`<...>` はそのまま OK (Jython テンプレートと同じ規約)。

| プレースホルダー | 意味 |
|---|---|
| `<分析名>` | 案件名 (例: 顧客解約予測 / 売上予測 / 店舗セグメンテーション) |
| `<INPUT_CSV_PATH>` | 入力 CSV 絶対パス |
| `<N>` / `<M>` | レコード件数 / 列数 |
| `<K>` | 分類クラス数 or クラスタ数 |
| `<ターゲット列名>` | 分類・数値予測のターゲット列名 |
| `<想定意思決定の要約>` | フェーズ1 で確認した想定意思決定 |
| `<モデル名1>` 〜 `<モデル名5>` | analysis HTML / PMML ファイル名から再構成したモデル表示名 |
| `<NN.NN>` | パーセンテージ / 小数の見本値 (実測の数値で置換) |
| `<クラスA>` `<クラスB>` | 分類クラスの実名 |
| `<フィールド名1>` 等 | 注目フィールドの実名 |
| `<YYYY-MM-DD>` | 生成日 |

---

## ヘッダー (タイトル + 前提メタデータ)

```markdown
# SPSS Modeler 分析サマリー — <分析名>

- 生成日: <YYYY-MM-DD>
- 入力データ: `<INPUT_CSV_PATH>` (<N> レコード × <M> 列)
- タスク種別: 分類 (<K> クラス)
- ターゲット: <ターゲット列名>
- 想定意思決定: <想定意思決定の要約> (確認済み)
```

タスク別の書き分け:
- 分類: `タスク種別: 分類 (<K> クラス)` / `ターゲット: <ターゲット列名>`
- 数値予測: `タスク種別: 数値予測` / `ターゲット: <ターゲット列名>`
- クラスタリング: `タスク種別: クラスタリング (<K> クラスタ)` / `ターゲット: なし`

---

## 1. 分析の目的と前提

```markdown
## 1. 分析の目的と前提

- **分析目的**: <分析目的を 1〜2 文で。何を予測・分類・分離したいか、何に使うか>
- **想定意思決定**: <想定意思決定の要約> (最終判断は <意思決定者>)
- **タスク種別**: 分類 (<K> クラス)
- **ターゲット変数**: <ターゲット列名> (採用根拠: <ユーザー明示 / 業務的に従属変数として妥当 / 等>)
- **置いた仮定**:
  - CSV 区切り文字: <カンマ / タブ / 等> / ヘッダ: <あり / なし> / 文字コード: <SystemDefault / UTF-8 / MS932>
  - Partition: 学習 <70>% / テスト <30>% (random_seed=<12345>)
- **対象モデル (<モデル数> 件)**: <モデル名1> / <モデル名2> / <モデル名3> / <モデル名4> / <モデル名5>
```

意思決定が未確認の場合:

```markdown
- **想定意思決定**: 未確認 (本レポートでは精度水準のみ評価する)
```

## 2. 分析結果の概要

走り読みできる結論を 4〜6 行のキーバリュー表で示す。

```markdown
## 2. 分析結果の概要

| 項目 | 値 |
|---|---|
| 最良モデル | <モデル名 (ベスト)> |
| テスト正解率 | <NN.NN>% |
| 学習-テスト乖離 | <NN.NN> pt |
| レコード数 (母集団) | <N> (学習 <N_train> / テスト <N_test>) |
| データ品質 (非欠損率) | <NN.N>% |

### 関連ファイル

- 入力サンプル: [sample_baseline.html](../outputs/sample_baseline.html)
- データ検査: [dataaudit_baseline.html](../outputs/dataaudit_baseline.html)
- モデル精度分析: [analysis_baseline.html](../outputs/analysis_baseline.html)
- 評価グラフ: [evaluation_baseline.html](../outputs/evaluation_baseline.html)
- 評価グラフ画像: [_tmp_evaluation_baseline.jpg](../outputs/_tmp_evaluation_baseline.jpg)
- 各フィールド分布画像: [_tmp_dataaudit_baseline_files/](../outputs/_tmp_dataaudit_baseline_files/)
- PMML XML: [../pmml/](../pmml/) (<モデル数> モデル)
- 生成 Jython: [../scripts/<basename>.py](../scripts/<basename>.py)
- 保存ストリーム: [../streams/<basename>.str](../streams/<basename>.str)
- 実行ログ: [../logs/clemb_execution.log](../logs/clemb_execution.log)
```

タスク別の指標差し替え:
- 分類: テスト正解率 / 学習-テスト乖離
- 数値予測: テスト R² (or 線型相関) / RMSE
- クラスタリング: クラスタ数 / Silhouette 等 (出力に出ていれば)

## 3. 分析結果から読み取れること

### 3.1 データの特徴 (dataaudit より)

```markdown
### 3.1 データの特徴

- レコード件数: <N> 件
- フィールド数: <M> (連続型 <a> / フラグ型 <b> / 名義型 <c>)
- 欠損: <なし / <フィールド名> に <NN>% の欠損あり 等>
- 注目フィールド:
  - **<フィールド名1>** (連続型): <最小値>〜<最大値>, 平均 <平均値> (歪度 <値>, <分布所見>)
  - **<ターゲット列名>**: <K> クラス分布 (<クラスA> が約 <NN>%, <クラスB> が <NN>%, 他 少数派)
  - **<フィールド名2> / <フィールド名3>**: <連続変数 / フラグ型 / 名義型>, <スケール所見>
- 個人情報候補: <無し / <疑わしいフィールド名> あり (3.1 と 6 章でフラグ)>
```

### 3.2 モデル比較 (analysis より、分類・数値予測のみ)

★BEST はテスト指標最良かつ学習-テスト乖離 ≤2 pt のものに 1 つだけ。決め手がなければ無印。

```markdown
### 3.2 モデル比較

| モデル | 学習 | テスト | 乖離 | 備考 |
|---|---|---|---|---|
| <モデル名1> ★BEST | <NN.NN>% | <NN.NN>% | <NN.NN> pt | <テスト最良。乖離許容範囲 / 等> |
| <モデル名2> | <NN.NN>% | <NN.NN>% | <NN.NN> pt | <汎化性能良好 / 等> |
| <モデル名3> | <NN.NN>% | <NN.NN>% | <NN.NN> pt | <過学習傾向 / 等> |
| <モデル名4> | <NN.NN>% | <NN.NN>% | <NN.NN> pt | <過学習傾向 / 等> |
| <モデル名5> | <NN.NN>% | <NN.NN>% | <NN.NN> pt | <やや精度低 / 等> |
```

数値予測の場合の例:

```markdown
| モデル | 学習 R² | テスト R² | 乖離 | 備考 |
|---|---|---|---|---|
| <モデル名 (ベスト)> ★BEST | <0.NNN> | <0.NNN> | <0.NNN> | <所見> |
| <モデル名2> | <0.NNN> | <0.NNN> | <0.NNN> | <所見> |
```

### 3.3 混同行列 / 残差傾向 (分類・数値予測のみ)

詳細表 (全モデルの一致行列) は `analysis_baseline.html` を参照する形にして、Markdown には所見だけ書く。

```markdown
### 3.3 ベストモデルの誤分類傾向

詳細は [analysis_baseline.html](../outputs/analysis_baseline.html) を参照。

<モデル名 (ベスト)> のテスト誤分類:
- <クラスA> → <クラスB> (<n> 件)
- <クラスA> → <クラスC> (<n> 件)

<少数クラス名 / 隣接クラス間 等> で誤分類が集中している。<クラス偏り / 特徴量分離が弱い 等> が影響している可能性がある。
```

数値予測の場合は「残差傾向」に置き換える:

```markdown
### 3.3 残差傾向

詳細は [analysis_baseline.html](../outputs/analysis_baseline.html) を参照。
予測値 -- 実測値の誤差は <概ね対称分布 / 正側にバイアスあり 等>。<フィールド名> が <大きい / 小さい> 領域で残差が大きくなる傾向あり。
```

### 3.4 評価グラフ所見 (分類のみ)

評価グラフは画像として埋め込む。読みは Gains チャートの一般論 + 過学習リスク。

```markdown
### 3.4 評価グラフ所見

![評価グラフ](../outputs/_tmp_evaluation_baseline.jpg)

上位 <NN>% で全正解の約 <NN>% を捕捉できており、ターゲットの絞り込み効果は <十分 / 限定的>。
ただし、テスト精度 <NN.NN>% は学習データへの依存度が <比較的高い / 低い> ($L 等で <NN>% 正解) ため、
未知データに対しては <数> ポイント程度の精度低下を想定して運用する。
```

クラスタリングの場合は 3.2〜3.4 を「クラスタ概況」1 章に置き換え:

```markdown
### 3.2 クラスタ概況

- クラスタ数: K = <K> (AutoCluster 自動決定)
- 主要分離変数: <フィールド名1>, <フィールド名2>
- 各クラスタのサイズ: C1: <n>件 / C2: <n>件 / ...
- BEST モデル: クラスタリングのため判定なし
```

## 4. ビジネスへの示唆

3 段落構成。各段落 2〜4 文。重要キーワードは `**bold**` で強調する。

```markdown
## 4. ビジネスへの示唆

**何が分かったか** — <モデル名 (ベスト)> がテスト <指標名> **<NN.NN>%** で最良の性能を示した。
<線形モデルが好成績 / 非線形モデルが優位 / 等> ということは、ターゲット (<ターゲット列名>) と入力変数の関係が <概ね線形分離可能 / 複雑な相互作用を含む / 等> であることを示唆している。
<次点モデル> も <NN.NN>% と <僅差 / やや差> で、<複雑な相互作用は限定的 / 等>。

**想定意思決定への適合度** — 想定意思決定 (<想定意思決定の要約>) に対しては、テスト <NN.NN>% は <十分な / 要検証な> 精度水準と評価できる。
ただし、<誤分類が少数クラスに集中 / 特定領域で残差が大きい 等> ため、本番運用前に <少数クラスの recall / 残差分布 / 等> を別途検証する必要がある。
<最終判断を補助する位置づけであれば MVP として運用可能 / 単独自動化には精度不足 / 等>。

**データ品質と運用上の留意点** — 母集団 <N> 件は <限定的 / 十分> で、<再現性確認のため追加データでの検証が望ましい / 等>。
<欠損や個人情報の混入は無く / 一部欠損あり / 等>、データ品質自体は <良好 / 要改善>。
<クラス偏り (<クラスA> が約 <NN>%) があるため少数クラスの予測根拠を ID 系変数に依存していないかを別途確認したい / 等>。
```

書き方ガイド:
- 1段落目「何が分かったか」: ベストモデル + 主要指標 + 業務ドメイン非依存の事実観察
- 2段落目「想定意思決定への適合度」: 意思決定が確認済みならその意思決定への精度の十分さ。**未確認なら「未確認のため精度水準のみ評価」と明記**
- 3段落目「データ品質と運用上の留意点」: クラス偏り / ID 系変数依存 / サンプルサイズ / 欠損 / 個人情報

## 5. 次に取るべきアクション

優先度高い順に 3〜5 個。各項目 1〜2 文。

```markdown
## 5. 次に取るべきアクション

1. **<少数クラス / 特定領域> の予測精度を確認する** — <recall / 残差分布 / 等> を個別に算出し、業務要件 (誤分類許容度) と照合する。
2. **追加データで再現性を検証する** — <別期間 / 別施設 / 別セグメント> データで同程度のテスト精度が出るか確認する。
3. **特徴量重要度を確認する** — <ベストモデル> の <係数 / Importance> を見て、業務担当が直感的に納得できる説明変数か検証する。
4. **(任意) 過学習傾向のあるモデル (<モデル名X> / <モデル名Y>) でハイパラ調整を試す** — テスト精度を改善できる余地があれば再評価する。
```

## 6. 確認ポイント

severity 3 段階で 4〜6 個。チェックリスト形式 (`- [ ]`)。

```markdown
## 6. 確認ポイント

- [ ] **(高)** サンプルサイズの妥当性 — <N> 件は <限定的 / 母集団に対し十分か未確認>。本番投入前に母集団の代表性を確認する。
- [ ] **(高)** ターゲット定義の妥当性 — <ターゲット列名> が <業務上の意味> として妥当か、業務側と認識合わせをする。
- [ ] **(中)** クラス偏り / データ偏りへの対処 — <クラスA / 特定セグメント> が約 <NN>% を占める。<少数クラスの recall / 偏りの影響> を個別に確認する。
- [ ] **(中)** 過学習傾向の検証 — <モデル名X> / <モデル名Y> は学習-テスト乖離 <NN> pt あり。本番運用には不向き。
- [ ] **(低)** PMML モデルの配備手順確認 — <モデル数> モデルの PMML XML が出力済み。本番環境への配備手順を確認する。
```

severity 選択基準:
- **高**: 本番投入を妨げる可能性がある (サンプルサイズ不足 / ターゲット定義の妥当性 / 個人情報混入)
- **中**: 業務適用前に検証が必要 (クラス偏り / 過学習 / 特徴量妥当性)
- **低**: 運用上の留意点 (配備手順 / 監視体制 / ドキュメント整備)
