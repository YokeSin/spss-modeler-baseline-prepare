# SPSS Modeler Baseline サマリーレポート生成 (フェーズ2)

このファイルは `SKILL.md`(フェーズ1: Jython 生成・実行)とペアで使う。
フェーズ1で `<RUN_DIR>` 配下に出力が揃った後に、本ファイルの手順で **Markdown サマリーレポート**を 1 本生成する。

## 目的

clemb 実行成功後に、以下を統合した **Markdown サマリーレポート**を `<RUN_DIR>/reports/summary_report.md` に出力する。

1. ユーザー入力から得た分析目的・タスク種別・想定意思決定
2. 実行後の出力 (DataAudit / Sample / Analysis / 評価グラフ HTML、PMML XML ファイル名) から導いた、入力データ概要・モデル構築結果・推奨アクション
3. AI による要約と確認ポイント

各章のスニペット例は [templates/section_examples.md](templates/section_examples.md) にある。**数値・表は AI が `outputs/` 配下の HTML を読んで Markdown 表に直書きする** (プレースホルダー置換は使わない)。詳細は元 HTML への相対リンクで誘導し、Markdown 側に重複データを大量に書かない方針。

## 完了条件

- `<RUN_DIR>/reports/summary_report.md` が生成されている (UTF-8、BOM なし)
- 以下 6 章のうち、タスク種別に該当するものが揃っている (見出しレベル `##`)
  1. 分析の目的と前提
  2. 分析結果の概要
  3. 分析結果から読み取れること (3.1〜3.4 のサブセクション)
  4. ビジネスへの示唆
  5. 次に取るべきアクション
  6. 確認ポイント
- ヘッダー直下のメタ情報 (生成日 / 入力データ / タスク種別 / ターゲット / 想定意思決定) が埋まっている
- 「2. 分析結果の概要」のキーバリュー表に結論判断に直接必要な値が入っている (例: 最良モデル / テスト精度 / レコード数 / データ品質)
- 「2. 分析結果の概要」末尾の **関連ファイル** リストに `outputs/` `pmml/` `scripts/` `streams/` `logs/` 配下への相対リンクが揃っている
- 「4. ビジネスへの示唆」に 3 段落 (何が分かったか / 想定意思決定への適合度 / データ品質と運用上の留意点) が入っている
- 「5. 次に取るべきアクション」が 3〜5 個ある
- 「6. 確認ポイント」が 4〜6 個あり、severity (**(高)** / **(中)** / **(低)**) 付き
- 分類タスクの場合: モデル比較表が 3.2 にあり、ベストモデルに `★BEST` が 1 つだけ付いている
- 分類タスクで評価グラフがある場合: 3.4 に `![評価グラフ](../outputs/_tmp_evaluation_baseline.jpg)` で画像が埋め込まれている
- 数値はすべて HTML 出力ファイルから読み取った実値である (推測値は使わない)
- HTML / CSS / `<style>` / `{{...}}` 形式のプレースホルダーが残っていない

## 対象範囲

- 既存の出力ファイル (`<RUN_DIR>/outputs/`, `<RUN_DIR>/pmml/`, `<RUN_DIR>/scripts/`, `<RUN_DIR>/streams/`, `<RUN_DIR>/logs/`) を読み取り、Markdown レポートを 1 本生成する
- `<RUN_DIR>/reports/` フォルダの作成
- 上記 6 章の Markdown 本文の直書き (スニペット例は [templates/section_examples.md](templates/section_examples.md))

## 対象外

- 新たな Modeler 実行やストリーム再構築 (必要なら `SKILL.md` 側に戻る)
- グラフ・図の新規作成 (`_tmp_evaluation_baseline.jpg` 等は `![](path)` で参照するのみ)
- プレゼン資料 (.pptx 等) の生成
- レポートの多言語化、PDF 変換 (Markdown レンダラーから印刷で代替)
- dataaudit の各フィールド分布画像 (7 枚以上) の Markdown 内埋め込み (重くなるためリンクで `dataaudit_baseline.html` へ誘導)
- 全モデルの混同行列のフル展開 (詳細は `analysis_baseline.html` を参照する形にする)

## 前提条件 (レポート生成を始める前のチェック)

以下が満たされていなければレポート生成を中断する。

- `<RUN_DIR>/logs/clemb_execution.log` が存在し、`[SPSS_JYTHON_FAILED]` 等のエラー文字列が含まれていない
- 必須出力が存在する
  - `<RUN_DIR>/scripts/<basename>.py`
  - `<RUN_DIR>/streams/<STREAM_FILENAME>`
  - `<RUN_DIR>/outputs/sample_baseline.html`
  - `<RUN_DIR>/outputs/dataaudit_baseline.html`
- タスク種別が分類または数値予測の場合: `<RUN_DIR>/outputs/analysis_baseline.html`
- タスク種別が分類の場合: `<RUN_DIR>/outputs/evaluation_baseline.html` と `<RUN_DIR>/outputs/_tmp_evaluation_baseline.jpg` (ファイル名は実物を Glob で確認)
- `<RUN_DIR>/pmml/` 配下に 1 個以上の XML (0 件なら確認ポイントに残す)

実際のファイル名は `outputs/` フォルダの実物を Glob で確認すること。Modeler 出力ノードの `output_name` 設定で別名 (`データ検査.html` 等) になっているケースがある。

不足がある場合は、レポート生成を試みる前に SKILL.md フェーズ1の作業手順 7. に戻って実行を完了させるか、ユーザーに確認する。

## 作業手順

### 1. 前提条件を確認する

- `<RUN_DIR>/logs/clemb_execution.log` を読み、エラー文字列が含まれていないことを確認する
- `outputs/` `pmml/` を Glob し、上記「前提条件」リストの必須出力が揃っていることを確認する
- 不足があれば中断し、ユーザーに状況を報告する

### 2. 入力ソースから値を抽出する

すべての出力は HTML (UTF-8 BOM 付き) で統一されている。HTML はそのまま Read で読み、`<TABLE>` `<TR>` `<TD>` 等のタグから値を抽出する (数値や見出しは目視で読める形式で出力されている)。HTML 全体を本文に転記しない。

抽出はセクション毎にメモする。Markdown 表で十分 (JSON 化は不要)。

#### 2.1 dataaudit (統計量 + 品質) — `<RUN_DIR>/outputs/dataaudit_baseline.html`

1 本の HTML 内に「統計量」表と「品質」表が連続している場合がある (`<TABLE>` が 2 つ並ぶ)。両方とも読む。

- 統計量列順 (Modeler 既定): 最小値 / 最大値 / 合計 / 集計範囲 / 平均値 / 平均の標準誤差 / 標準偏差 / 分散 / 歪度 / 歪度の標準誤差 / 尖度 / 尖度の標準誤差 / 中央値 / 最頻値 / カテゴリ数 / 有効
- 品質列順 (Modeler 既定): 外れ値 / 極値 / アクション / 欠損値の代入 / 代入方法 / 非欠損値の割合(%) / 有効なレコード / ヌル値 / 空文字列 / 空白文字 / 空白値
- 「有効」列の最大値を「件数 (母集団)」とする
- 各フィールドの分布画像は `_tmp_dataaudit_baseline_files/_tmp_dataaudit_baseline_propN.jpg` (Markdown には埋め込まずリンクのみ)
- 「尺度」列の値で連続型 / フラグ型・順序型 / 名義型 を見分ける (3.1 のメモに使う)

Markdown レポートには全フィールドの統計量を表で書かない (元 HTML に詳細表があるため)。代わりに 3.1「データの特徴」で注目フィールド 3〜5 個に絞って文章で記述する。

#### 2.2 sample — `<RUN_DIR>/outputs/sample_baseline.html`

- 先頭の `<TR>` がヘッダ。列名・列数を把握する
- 各データ行の値を見て、個人情報候補 (氏名・住所・電話番号・メールアドレス) があれば 3.1 と 6 章 (確認ポイント) でフラグを立てる
- Markdown レポートにはサンプル表は書かない (元 HTML を [sample_baseline.html](../outputs/sample_baseline.html) リンクで誘導するだけ)

#### 2.3 analysis (分類・数値予測のみ) — `<RUN_DIR>/outputs/analysis_baseline.html`

- モデル毎に
  - `$X-<target> と <target> の比較` ヘッダの直後 `<pre>` から `正解` 行の `1_学習` 列・`2_テスト` 列の % を読む → モデル別 train/test 正解率
  - `$X-<target> の一致行列` ブロックから 2 つの `<pre>` (学習 / テスト) を読み、混同行列を把握する (Markdown には全数値を書かず、ベストモデルの誤分類傾向だけ 1〜2 文で書く)
  - 1 つ目の `<pre>` のヘッダ列 (`薬品A 薬品B ...`) からクラス名一覧を再構成
  - 末尾の `$L-... $N-... 間の一致` セクションは集計用なのでレポート対象外
- モデルの接頭辞 (`$L`, `$N`, `$XGT`, `$R`, `$R1`, `$XS`, `$XS1` 等) は analysis HTML に出てくるものをそのまま使う

#### 2.4 数値予測 / クラスタリングの場合

- 数値予測: 3.2 のモデル比較表は「学習 R² / テスト R²」等に置き換え。3.3 は「残差傾向」に変更、3.4 (評価グラフ) は削除
- クラスタリング: analysis HTML はそもそも存在しないため、3.2〜3.4 を「3.2 クラスタ概況」1 章に置き換える。BEST バッジなし

#### 2.5 PMML ファイル名 — `<RUN_DIR>/pmml/*.xml`

- ファイル名 (例: `02_ニューラル_ネット_1.xml`) からモデル表示名を再構成する: 数値プレフィックス除去 + `_` をスペースに戻す → `"ニューラル ネット 1"`
- PMML 本体は読まない (重い、内容も AI が解釈しづらい)

#### 2.6 抽出値の中間メモ形式

無駄に詳細にしない。Markdown 表 1 つくらいで十分。

```
| フィールド | 尺度 | 最小値 | 最大値 | 平均 | 有効 | 欠損(%) |
| 年齢 | 連続型 | 15 | 74 | 44.3 | 200 | 0 |
| 性別 | フラグ型 | -- | -- | -- | 200 | 0 |
```

```
| モデル | code | 学習% | テスト% | 乖離 | best |
| ロジスティック回帰 | $L | 100.00 | 97.22 | 2.78 | YES |
| ニューラル ネット | $N | 96.09 | 94.44 | 1.65 |  |
```

### 3. Markdown レポートを組み立てる

#### 3.1 ファイルを新規作成する

```text
<RUN_DIR>/reports/summary_report.md
```

`reports/` フォルダがなければ作成する。既存の `summary_report.md` がある場合は上書きする (版管理が必要なら別 run フォルダで再実行する)。

#### 3.2 章別に本文を直書きする

[templates/section_examples.md](templates/section_examples.md) を参照しながら、`outputs/` 配下の HTML から読んだ値を Markdown 表 / 段落で書き下ろす。

書き順は以下:

a. **ヘッダー** (タイトル + メタ情報リスト)
   - `# SPSS Modeler 分析サマリー — <分析名>`
   - 生成日 (YYYY-MM-DD) / 入力データパス + 件数 + 列数 / タスク種別 / ターゲット / 想定意思決定 を `-` リストで

b. **1. 分析の目的と前提**
   - 分析目的 / 想定意思決定 / タスク種別 / ターゲット変数とその根拠 / 置いた仮定 / 対象モデル一覧 を `-` リストで
   - 想定意思決定が未確認なら「未確認 (本レポートでは精度水準のみ評価する)」と明記

c. **2. 分析結果の概要**
   - 4〜6 行のキーバリュー表 (`| 項目 | 値 |`)
   - `### 関連ファイル` サブセクションに `outputs/` `pmml/` `scripts/` `streams/` `logs/` 配下の相対リンクを `-` リストで列挙
     - 相対パスは `../outputs/...` `../pmml/...` 形式 (`<RUN_DIR>/reports/` からの相対)
     - リンク表記: `[ファイル名](../outputs/ファイル名)` または `[人間可読ラベル](../outputs/ファイル名)`
   - 分類のみ: `analysis_baseline.html` `evaluation_baseline.html` `_tmp_evaluation_baseline.jpg` を含める
   - 数値予測のみ: `analysis_baseline.html` を含める (評価グラフ系は省略)
   - クラスタリングのみ: `analysis_baseline.html` `evaluation_baseline.html` は省略

d. **3. 分析結果から読み取れること**
   - **3.1 データの特徴** (dataaudit より): 件数 / 欠損率 / 注目フィールド 3〜5 個 / 個人情報候補有無
   - **3.2 モデル比較** (分類・数値予測のみ): モデル数ぶんの行を持つ Markdown 表 (モデル / 学習 / テスト / 乖離 / 備考)。ベストモデルにだけ `★BEST` バッジを名前の後ろに付ける
   - **3.3 ベストモデルの誤分類傾向 / 残差傾向** (分類・数値予測のみ): 2〜3 文。詳細は `analysis_baseline.html` リンクで誘導
   - **3.4 評価グラフ所見** (分類かつ評価グラフあり): `![評価グラフ](../outputs/_tmp_evaluation_baseline.jpg)` で画像埋め込み + 2〜4 文の所見
   - クラスタリングは 3.2〜3.4 を「3.2 クラスタ概況」1 章に置き換える

e. **4. ビジネスへの示唆**
   - 3 段落構成 (各段落 2〜4 文)。書き方は §AI 要約の書き方ガイド
   - 重要キーワードは `**bold**` で強調

f. **5. 次に取るべきアクション**
   - 番号付きリスト 3〜5 個。各項目 1〜2 文。優先度高い順
   - キーワードは `**bold**`

g. **6. 確認ポイント**
   - チェックリスト形式 (`- [ ]`) で 4〜6 個
   - 各項目に severity を `**(高)**` / `**(中)**` / `**(低)**` で頭に付ける
   - severity 基準は [templates/section_examples.md](templates/section_examples.md) §確認ポイント末尾参照

#### 3.3 タスク種別による不要セクションの削除

- **タスクが数値予測** の場合:
  - 3.4 (評価グラフ所見) を削除
  - 「2. 関連ファイル」から `evaluation_baseline.html` `_tmp_evaluation_baseline.jpg` の行を削除

- **タスクがクラスタリング** の場合:
  - 3.2〜3.4 を「3.2 クラスタ概況」1 章に置き換え
  - 「2. 関連ファイル」から `analysis_baseline.html` `evaluation_baseline.html` `_tmp_evaluation_baseline.jpg` を削除
  - 「2. 概要」のキーバリュー表からテスト正解率行を削除し、クラスタ数 / 主要分離変数 等に変更
- **タスクが分類だが評価グラフが存在しない** 場合:
  - 3.4 ブロックだけを削除
  - 「2. 関連ファイル」から `evaluation_baseline.html` `_tmp_evaluation_baseline.jpg` の行を削除

#### 3.4 ファイル保存

- `<RUN_DIR>/reports/` フォルダがなければ作成する
- 文字コード: **UTF-8 (BOM なし)** で保存する
  - Markdown レンダラーは BOM があると先頭に文字化けが出る場合がある。BOM なしを既定とする

## AI 要約の書き方ガイド (4. ビジネスへの示唆)

3 段落構成 + 「5. 次に取るべきアクション」で構成する。

| 段落 | 内容 | ソース |
|---|---|---|
| 段落1 | 何が分かったか (ベストモデル + テスト指標) | analysis HTML |
| 段落2 | 想定意思決定への適合度 (業務ドメイン非依存の汎用評価に留める。意思決定が未確認なら「未確認のため精度水準のみ評価」と書く) | フェーズ1 の意思決定 + analysis |
| 段落3 | データ品質や運用上の留意点 (クラス偏り・ID 系変数依存・サンプルサイズの限界等) | dataaudit + 母集団件数 |

`**...**` で囲んだ語は太字で強調表示される。重要数値や結論キーワードに使うとよい。
段落の頭に `**何が分かったか** — ` のようにラベルを置くと読みやすい。

## 重要ルール

- **数値は HTML ファイルから読み取った値だけを書く**。推測値は使わない。読めなかった値は「-」または `(取得失敗)` のように明示する
- **「4. ビジネスへの示唆」3 段落 + 「5. 次に取るべきアクション」3〜5 個 は必ず埋める**。読み手が走り読みできる結論を冒頭に置く
- **2 段落目「想定意思決定への適合度」は業務ドメイン非依存の汎用評価に留める**。想定意思決定がフェーズ1 で確認できている場合のみ、その意思決定への適合度に言及する (未確認の場合は「未確認のため精度水準のみ評価」と明記)
- **HTML / CSS / `<style>` / `<table>` 等の HTML タグを Markdown に持ち込まない**。表は Markdown のパイプ表記 (`|`) で書く
- **プレースホルダー (`{{...}}`) を作らない**
- **Python による文字列置換ワークフローを使わない**。AI が Read で値を読み、Write で直接 Markdown を書く
- **ベストモデルの判断 (分類)**: テスト正解率が最大、かつ学習-テスト差が小さいものに `★BEST` を付ける。差が大きい (例: 学習 100% / テスト 91.67% で乖離 8 pt 超) 候補は基本選ばない。クラスタリングは BEST バッジを付けない
- **モデル名は `<RUN_DIR>/pmml/` のファイル名から再構成する** (例: `02_ニューラル_ネット_1.xml` → `"ニューラル ネット 1"`)。PMML XML 本体は読まない
- **個人情報を疑わせる項目** (氏名・住所・電話番号・メールアドレス等) はサマリーに含めない。dataaudit / sample に出ていた場合は 3.1 と 6 章 (確認ポイント) でフラグだけ立て、具体的な値は書かない
- **画像埋め込みは評価グラフ 1 枚のみ** (`![評価グラフ](../outputs/_tmp_evaluation_baseline.jpg)`)。dataaudit の各フィールド分布画像は埋め込まずリンク (`[dataaudit_baseline.html](../outputs/dataaudit_baseline.html)`) で誘導する
- **全モデルの混同行列フル展開はしない**。ベストモデルの誤分類傾向だけ 1〜2 文で書き、詳細は `analysis_baseline.html` リンクで誘導する
- **相対リンクは `<RUN_DIR>/reports/` を起点とする**。`../outputs/...` `../pmml/...` 形式
- **`reports/` フォルダは skill 側で作成する** (Jython テンプレートは作らない)
- **既存の `summary_report.md` がある場合は上書きする** (版管理が必要なら別 run フォルダで再実行する)

## 生成後チェックリスト

レポートをユーザーに渡す前に、以下を確認する。

- [ ] `<RUN_DIR>/reports/summary_report.md` が存在し、サイズが 0 バイトでない
- [ ] UTF-8 BOM なしで保存されている
- [ ] HTML タグ (`<style>` `<table>` `<tr>` `<td>` `<div>` 等) が Markdown 本文に混入していない (画像埋め込みの `![](...)` 以外)
- [ ] `{{...}}` プレースホルダーが残っていない
- [ ] 6 章の見出し (`## 1.` 〜 `## 6.`) が揃っている (タスク種別に応じてサブセクションが入れ替わっているのは OK)
- [ ] ヘッダー直下のメタ情報 (生成日 / 入力データ / タスク種別 / ターゲット / 想定意思決定) が埋まっている
- [ ] 「2. 概要」のキーバリュー表に 4〜6 行ある
- [ ] 「2. 関連ファイル」リストの相対リンクがすべて実ファイルを指している (`outputs/` `pmml/` `scripts/` `streams/` `logs/`)
- [ ] 「4. ビジネスへの示唆」に 3 段落入っている
- [ ] 2 段落目「想定意思決定への適合度」が業務ドメイン非依存の汎用評価に留まっている
- [ ] 「5. 次に取るべきアクション」が 3〜5 個ある
- [ ] 「6. 確認ポイント」が 4〜6 個あり、severity (`**(高)**` / `**(中)**` / `**(低)**`) 付き
- [ ] 分類タスクの場合、3.2 のモデル比較表に `★BEST` が 1 つだけ立っている
- [ ] 分類タスクで評価グラフがある場合、3.4 に `![評価グラフ](../outputs/_tmp_evaluation_baseline.jpg)` の画像埋め込みがある
- [ ] 数値はすべて出典 HTML から読み取った値である (推測なし)
- [ ] ターゲット変数とタスク種別が SKILL.md フェーズ1 で確定したものと一致
- [ ] 数値予測タスクの場合、3.4 (評価グラフ) が削除されている
- [ ] クラスタリングタスクの場合、3.2〜3.4 が「3.2 クラスタ概況」に置き換わっている
- [ ] 個人情報候補の値がベタ書きされていない
